VULNERABILITIES

What is a vulnerability?

A vulnerability is a weakness of a system which can be exploited by a threat actor to perform unauthorized actions within the system. Some vulnerabilities are directly tied with bugs in the code, others are an innate repercussion of the type of the software, and others still are the result of a lack of implementation of mitigation strategies. What they all have in common is that they're not a security hazard by themselves, but only when a third party exploits them.

In this section you will find list and describe some of the most harmful vulnerabilities of software that can still be found in software today. In addition, every vulnerability described will also provide information on the possible consequences of that vulnerability being exploited as well as strategies to detect and mitigate that vulnerability to help making your software more secure. You will also be able to see the effects of these exploits by visiting related articles in the History section.

Articles

Uncontrolled resource consumption

The software does not check the amount of energy and resources consumed, which can lead to DoS attacks.

Improper input validation

Inputs are not controlled or can be mistaken by the software for a different type of input.

Information exposure

The private information of users in unintentionally exposed to threat actors.

Improper restriction of operations

The software performs operations outside of its intended memory buffer, leading to data leaks.

Incorrect permission assignment

Permissions are assigned in such a way that important resources are able to be read or modified by threat actors.

Improper authentication

The software doesn't check the identity of users sufficiently enough, allowing threat actors to make incorrect claims.

Definitions

In order to better understand each vulnerability, certain recurring key words will be described here in the homepage. Clicking on a red term in one of the articles will bring you here to its respective definition.

Types

The type of a vulnerability represents its level of abstraction, meaning the range of situations, software types, and programming languages where that vulnerability can appear. For example a vulnerability that can only appear in a specific programming language is less abstract than one that can appear in any software.

For more information visit the CWE website.

C

Class type

Vulnerabilities of the "class" type are wide-spread accross all types of software. They are more general than "base" types and they can sometimes include more "base" and "variant" vulnerabilities within them. For example, Improper Restriction of Operation is a type C vulnerability that includes Out of Bounds Read and Write, two type B vulnerabilities.

B

Base type

A "base" type vulnerability is described in an abstract fashion, however with sufficient details to be able to apply specific and targeted ways of detection and prevention. They are more general than "variant" types but more specific than "class" types. They can sometimes include multiple "variant" types within them.

V

Variant type

a "variant" type vulnerability is often only found either in a specific subset of software or in software written in a specific programming language. Because of this, they are more specific than "base" weaknesses, and can often be associated with one or more other "variant" type vulnerabilities that exploit related issues. For example, the type B vulnerability Out of Bounds Read includes two related type V vulnerabilities, Buffer Over-read and Under-read.

Consequences

The danger of vulnerabilities comes from the various consequences that occur when they are exploited. The following definitions describe some of the most dangerous consequences of software vulnerabilities. Each vulnerability can lead to multiple consequences, visit the articles section for more information.

Denial of Service (DoS)
DoS is one of the most common consequences of software vulnerabilities, it consints in disrupting a machine or program by flooding its input with useless requests that overload the system preventing legitimate requests from being fulfilled. This can lead to a crash of the system.
Fail Opening
"Fail Closed" software is software that is able to block access to users when exceptions are encountered. If a threat actor is able to "Fail Open" a system, then that system will be able to grant access even when exceptions are met. For example, an online market will grant access to its products when payment does not terminate sucsessfully.
Memory Access
If the threat actor is given the ability to modify or overwrite data in the memory, it could be able to change important flags, such as permissions of users, that could lead to more dangerous consequences.
Data Corruption
If the threat actor is unable to directly control memory, but is still able to change it, such as through an overflow, it is considered data corruption. Data corruption can vary in size from a minor loss of data to a change that makes the affected memory portion unable to be identified or computed by the program, resulting in the program erroring or crashing.

Detection

The following terms describe several ways of detecting possible vulnerabilities in a code. Certain solutions are better suited for certain vulnerabilities, visit the articles section for more information.

Static Analysis
This debugging method consists in reading the source code of a program without executing it to find errors and spot vulnerabilities. Carrying out this method requires a base understanding of the code structure. Certain automated tools can help programmers in this task.
Dynamic Analysis
The evaluation of a program is carried out by executing it and looking at the data in real time, while the program is running. It is important that a program is tested in all scenarios possible, as to reduce the risk of overlooking a fatal flaw.
Fuzzing
The program is given invalid, unexpected or random data as input, and is then monitored for crashes, memory leaks or other exceptions. This process is often automated and is the most effective when the program recievies semi-valid inputs, which exposes issues regarding corner cases.
Spotchecking
This test is similar to static analysis, however it is only performed in a small percentage of spots of the source code, mainly the most vulnerable parts.
Manual Code Review
The process of reading through the code line by line to spot errors or vulnerabilities. Ideally, all of the programmers review the entirety of the source code in order to find as many errors as possible.
Software Review
A process of meeting during which a software product is examined by a project personnel, managers, users, customers, user representatives, or other interested parties for comment or approval. IEEE Std 1028 is a common set of activities for formal reviews with the objective of identifying issues.

Mitigation

There are many possible ways of eliminating or mitigating the effect of a vulnerability. Certain solutions will work best for certain vulnerabilities while others may be not effective, visit the articles section for more information.
