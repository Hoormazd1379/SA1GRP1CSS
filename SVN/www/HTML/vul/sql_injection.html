Vulnerabilities

SQL Injection

Statistics

  • Type: Base

  • CWE Number: 89

  • Applicable Platforms: Database server

  • Likelyhood of exploit: High

Description

SQL Injection is the practice of introducing malicious code inside an SQL server that is then executed on the machine. Using this method, it is possible to cause serious damage to a database.

This is possible because vulnerable systems can accept SQL command syntax as input, because the system either doesn't neutralize or doesn't quote its inputs. Because of that, those inputs will be interpreted as SQL instead of ordinary user data, which allows threat actors to alter query logic, bypass security checks, or modify the database, as, by default, SQL does not make a distinction between the data plane and the control plane.

Because of the nature of this exploit, it is easy to attempt an attack on any SQL server, leaving vulnerable servers very often identified very quickly.
SQL Injection can be classified under Improper Input Validation.

What is SQL?

SQL, which stands for "Structured Query Language", is a domain-specific language used and designed for managing large databases, and thus it is in close contact with large amounts of data.
It is particularly useful in handling structured data, which is made up of relations between different types of data, such as variables.

Consequences

Common consequences of SQL Injection include loss of integrity and loss of confidentiality.

In some cases, either if authorization information is stored in the database and are able to be changed via SQL injection, or if the system uses SQL commands in order to validate user names and passwords allowing threat actors to log in as a user without knowing the password, access control is compromised.

Detection

Detection of this vulnerability is not as easy as automated static analysis which might not be able to detect the usage of third-party libraries or API functions that indirectly invoke SQL commands.

Manual code reviews can prove useful, however they are more time-consuming than spotchecking, which only covers a small portion of the code and can often not be enough as a result.

Mitigation

If the set of acceptable inputs is limited or known, it may prove useful to assign every valid input a numerical value, such as a numeric ID, and reject all other inputs.

Another strategy is to, if available, use structured mechanisms that automatically enforce the separation between data and code. These mechanisms may be able to provide the relevant quoting, encoding, and validation automatically, instead of relying on the developer to provide this capability at every point where output is generated.
