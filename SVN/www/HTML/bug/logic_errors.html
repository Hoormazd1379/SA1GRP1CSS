Logic Error

The bugs of this type are connected to the program's operating logic and are the most difficult to identify; in fact if you want to compile a program, the compiler doesn't identify any errors, from the moment it doesn't have any reason to report them since the program is semantically and syntactically correct; therefore doesn't mean that our program is "logically" correct, in fact in many cases the end result doesn't match what we expected, so it is up to the programmer to locate them.

They can be caused by a lack of understanding of the problem to be calculated or the constraints that the input data must respect, or even in the evaluation of the algorithm test. These errors typically lead to the design of an algorithm that does not provide the required output within one or more instances of the problem to be solved and we need to use the debugger to find it.

A fairly trivial example is when we divide a number by zero or when we use a variable,which we haven't defined it before, syntactically is correct, but we have the error during the execution of the program, which stops.

OBOE

An OBOE, which means off-by-one error, or also commonly known as an OBOB (off-by-one bug), is a logic error involving the discrete equivalent of a boundary condition. It often occurs in computer programming when an iterative loop iterates one time too many or too few. This problem could arise when a programmer makes mistakes such as using "is less than or equal to" where "is less than" should have been used in a comparison, or fails to take into account that a sequence starts at zero rather than one (as with array indices in many languages). This can also occur in a mathematical context.

Fencepost errors

A fencepost error is a specific type of an off-by-one-error.

We can understand better if we take an example: consider a fence 30 meters long and every 3 meters you have to put a pole, the question that arises is: how many poles do we need?

Instantly we come to answer 10, but it's wrong since we have 11 seats.

If we want to broaden and generalize the concept; the problem can be declared as follows: if you have n places, how many sections are there between them?

There are three possibilities:

the answer can be n-1 if you have an open-ended posts-line, while it will be n if the posts-line forms a circle and finally we have n + 1 if the messages are not at the end of the barrier. Thus the Fencepost errors come from the inaccurate counting of things either spaces between them or even if one or both ends should be counted.

Fencepost error may refer to an error caused by unexpected regularities in input values, which could completely counter a function. This error involves the difference between expected behaviors and worst cases of an algorithm. In these cases the accuracy is essential to avoid committing an OBOE that could lead to disastrous effects.

Infinite loops and infinite recursion

The difference between an infinite loop and an infinite recursion is that the infinite recursion is a special case of an infinite loop; then the question arises what is an infinite loop?

An infinite loop (or endless loop) is a sequence of instructions in a computer program which loops endlessly, either due to the loop having no terminating condition (having one that can never be met) or one that causes the loop to start over. In older operating systems with cooperative multitasking, infinite loops normally caused the entire system to become unresponsive. With the now-prevalent preemptive multitasking model, infinite loops usually cause the program to consume all available processor time, but can usually be terminated by the user. Busy wait loops are also sometimes called "infinite loops", which can cause for a computer "freezing"; others include thrashing, deadlock, and access violations. Instead the recursive loop refers to a method of defining the functions in which the function to be defined is applied within their own definition.
